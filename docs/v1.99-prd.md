# v1.99 Product Requirements Document
## Performance & Learning Integrity Layer (PLIL)

**Release branch:** `release/v1.99`
**Status:** Planning
**Date:** 2026-04-02
**Depends on:** v1.97 shipped; v1.98 in flight or complete

---

## Problem Statement

The system cannot answer the only question that matters for a probabilistic decision engine:

> **Is Thufir's probability estimate better than the market's implied probability?**

Two structural defects prevent this question from being answered today.

**Defect 1 — Probability contamination.**
The two primary prediction write sites (`src/execution/modes/paper.ts:97`, `src/execution/modes/webhook.ts:24`) store the market's implied price as `predicted_probability`. This means `brier_contribution`, calibration reports, and learning weight updates are all computed against the market's own estimate — not Thufir's. The system appears to be measuring model quality; it is measuring market efficiency.

**Defect 2 — Untrustworthy outcome labels.**
The resolver (`src/core/resolver.ts`) marks a prediction as resolved when the market's YES price crosses 0.5. A snapshot price is not a truth label. Training on these labels teaches the model to track interim market sentiment, not to predict final outcomes.

Both defects produce the same failure mode: a system that generates plausible-looking metrics while the underlying learning signal is noise. Fixing these is a prerequisite to every other improvement in this layer.

---

## Goals

### Primary (Phase 1) — Data Integrity
Make every prediction record carry a clean separation between what Thufir believed and what the market implied at the time. Ensure outcome labels only reflect confirmed final resolution.

1. Add `model_probability` (Thufir's raw estimate) and `market_probability` (market-implied price at decision time) as distinct columns in `predictions`.
2. Fix the two write sites to populate both fields correctly.
3. Add `outcome_basis` column: `'final'` (market resolved) or `'estimated'` (snapshot threshold). Exclude `'estimated'` rows from all learning pipelines.
4. Mark all pre-v1.99 rows as `'legacy'` — excluded from metrics by default, queryable for historical audit only.

### Secondary (Phase 2) — Metrics Engine
Once 50+ `final`-resolution predictions exist with clean fields, build the measurement layer.

5. Create `learning_examples` as a SQL view over the cleaned `predictions` table — single source of truth for metrics and learning.
6. Rolling window metrics (last 10/20/50/100/200 final predictions): accuracy, model Brier, market Brier, Brier delta, avg edge, PnL.
7. Wire Brier delta into the entry gate as a hard domain-level execution threshold.
8. Surface rolling windows on the existing dashboard Performance tab (no new tab required — extend what exists).

---

## Non-Goals

- New trading strategies or signal sources
- Calibration scalar applied at inference time (this comes after Phase 2 produces enough data to fit a reliable scalar — minimum ~200 per domain)
- Drift detection automation (manual inspection sufficient until sample sizes justify it)
- Backfilling `model_probability` on legacy rows — we do not know what Thufir's estimate actually was; do not fabricate it
- Replacing the v1.97 entry gate logic

## Architecture

### Before (current state)

```
Originator proposes trade
  → write site stores market.prices[outcome] as predicted_probability   ← contaminated
  → resolver marks resolved when YES price ≥ 0.5                       ← unreliable
  → brier_contribution computed on contaminated probability              ← noise
  → calibration report reads contaminated rows                          ← meaningless
  → learning.ts weight update reads contaminated rows                   ← teaches nothing
  → entry gate ignores all of the above                                 ← loop not closed
```

### After (v1.99)

```
Originator proposes trade
  → model_probability = Thufir's LLM estimate                           ← clean
  → market_probability = market.prices[outcome] at decision time        ← clean
  → resolver marks 'final' only on confirmed market resolution          ← trustworthy
  → learning_examples VIEW selects only final + non-legacy rows         ← clean dataset
  → rolling window query over learning_examples                         ← real metrics
  → entry gate reads domain Brier delta, blocks if below threshold      ← loop closed
```

## Data Model

### Changes to `predictions` table

```sql
ALTER TABLE predictions ADD COLUMN model_probability REAL;
ALTER TABLE predictions ADD COLUMN market_probability REAL;
ALTER TABLE predictions ADD COLUMN outcome_basis TEXT DEFAULT 'legacy'
  CHECK (outcome_basis IN ('final', 'estimated', 'legacy'));
```

**Semantics:**
- `model_probability` — the probability Thufir assigned to the predicted outcome, as output by the LLM or decision logic. Never derived from market prices.
- `market_probability` — `market.prices[decision.outcome]` at the moment of prediction creation. Represents what the market implied at decision time.
- `outcome_basis`:
  - `'final'` — market fully resolved, outcome is confirmed truth
  - `'estimated'` — outcome inferred from snapshot threshold; never used in learning
  - `'legacy'` — pre-v1.99 row; excluded from metrics by default

### `learning_examples` view

```sql
CREATE VIEW learning_examples AS
SELECT
  id,
  domain,
  regime_tag        AS regime,
  strategy_class,
  model_probability,
  market_probability,
  executed,
  position_size,
  CASE WHEN outcome = 'YES' THEN 1 ELSE 0 END AS outcome_value,
  pnl,
  (model_probability  - CASE WHEN outcome = 'YES' THEN 1.0 ELSE 0.0 END) *
  (model_probability  - CASE WHEN outcome = 'YES' THEN 1.0 ELSE 0.0 END) AS brier_model,
  (market_probability - CASE WHEN outcome = 'YES' THEN 1.0 ELSE 0.0 END) *
  (market_probability - CASE WHEN outcome = 'YES' THEN 1.0 ELSE 0.0 END) AS brier_market,
  created_at,
  outcome_timestamp  AS resolved_at
FROM predictions
WHERE outcome_basis = 'final'
  AND model_probability IS NOT NULL
  AND market_probability IS NOT NULL
  AND outcome IS NOT NULL;
```

## Metrics Definition

All metrics computed over `learning_examples`.

| Metric | Formula | Interpretation |
|---|---|---|
| Accuracy | correct predictions / N | Binary. Secondary metric only. |
| Brier (model) | avg((model_probability − outcome)²) | Lower = better. Primary metric. |
| Brier (market) | avg((market_probability − outcome)²) | Baseline. What a passive observer achieves. |
| Brier delta | brier_market − brier_model | **Positive = Thufir adds value.** Negative = worse than market. |
| Avg edge | avg(model_probability − market_probability) | Structural edge claimed at decision time. |

**Minimum sample threshold before any metric is reported: 20 final predictions per window.**
Below this, return `null` — never show a metric computed on fewer than 20 samples.

## Policy Integration

The entry gate (`src/core/llm_entry_gate.ts`) reads domain-level Brier delta before approving execution.

Rules (deterministic, not LLM-driven):
```
if domain has < 20 final predictions:
  allow with no calibration adjustment (insufficient data)

if brier_delta(domain, last_50) < 0 AND sample_count >= 50:
  block execution for this domain
  log reason: 'domain_calibration_below_market'

if brier_delta(domain, last_20) < -0.05 AND sample_count >= 20:
  reduce position_size by 50%
  log reason: 'domain_calibration_degrading'
```

These rules augment the existing gate logic; they do not replace it.

## Success Criteria

Phase 1 is complete when:
- Every new prediction row has non-null `model_probability` and `market_probability` with distinct values
- No new rows receive `outcome_basis = 'estimated'` or `'legacy'`
- The resolver produces `outcome_basis = 'final'` only on confirmed market resolutions
- All existing calibration and learning queries work correctly on the cleaned schema

Phase 2 is complete when:
- `/api/dashboard` returns rolling window metrics in the performance section
- Entry gate reads and applies domain Brier delta rule
- A domain blocked by the calibration gate produces a logged rejection with reason `'domain_calibration_below_market'`

## Risks

| Risk | Mitigation |
|---|---|
| Thufir's model_probability is not always recoverable (LLM output not parsed) | Fail prediction creation; never store a row without model_probability if the write site has access to the estimate |
| Small sample Brier delta is noisy and blocks good trades | Hard minimum of 20 samples before rule fires; 50 samples before domain block |
| Legacy rows contaminate metrics if join is wrong | `outcome_basis = 'legacy'` filter is in the view definition, not in ad-hoc queries |
| v1.97 gate already running — schema migration must not break it | Migration is additive only; new columns nullable; existing code paths unaffected |
