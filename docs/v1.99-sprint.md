## Sprint Goal

Make every prediction row carry a clean, auditable separation between what Thufir believed and what the market implied. Ensure outcome labels are trustworthy. Close the loop from clean data to gate enforcement.

This sprint has two phases. Phase 2 does not start until Phase 1 has produced 50+ `final`-resolution predictions in the production DB.

**Updated 2026-04-03:** Production DB audit found three structural blockers that must be resolved before Phase 1 can produce meaningful data. Added as Phase 0.

## Phase 0 — Prerequisites (discovered 2026-04-03, do first)

These three gaps mean Phase 1 would ship clean schema with no data flowing through it. Fix them first.

### Task 0a — `feat/v1.99-originator-execution-fix`
**Trace and fix the gate approval → execution gap**

**Branch base:** `release/v1.99`

**Context:** Production data shows 2 originator gate approvals, 0 executions, `llm_trade_proposals.executed` always 0. Something between `LlmEntryGate` returning `approve` and the trade fill is blocking. Until this is fixed, `model_probability` is never written to `predictions` (no originator trade = no prediction row = Phase 1 produces nothing).

**Changes:**
- Trace the code path in `src/core/autonomous.ts` from `entryGate.evaluate()` returning `approve` through to the actual order placement; identify what blocks execution
- Fix the blocker (likely a missing funds check, a size calculation producing zero, or a guard that rejects the proposal before the order call)
- Call `updateTradeProposalOutcome(symbol, side, true, tradeId)` after a confirmed fill on the originator path — this function exists in `src/memory/llm_trade_proposals.ts` but has zero call sites

**Tests required:**
- Mock gate returning `approve` for an originator candidate → trade executes, `llm_trade_proposals.executed = 1`
- Mock gate returning `reject` → no execution, `executed` remains 0

**Done when:** An originator proposal approved by the gate results in a paper fill and `executed = 1` in `llm_trade_proposals`.

### Task 0b — `feat/v1.99-probability-threading`
**Thread originator `confidence` as `modelProbability` to prediction write sites**

**Branch base:** `release/v1.99`
**Depends on:** Task 0a (originator must execute before this matters)

**Context:** TDD §2.4 assumes `decision.probability` exists on the candidate object reaching `createPrediction`. It doesn't — `TradeProposal` uses `confidence: number`. The field exists under a different name; nobody has mapped it. Until this is fixed, `model_probability` is always `undefined` on new prediction rows even after Task 0a ships.

**Changes:**
- `src/core/autonomous.ts` — when building the gate candidate from an originator proposal, map `proposal.confidence` → `candidate.modelProbability` (or rename field; pick one and be consistent)
- `src/execution/modes/paper.ts` — confirm `candidate.modelProbability` reaches `createPrediction({ modelProbability: ... })`
- `src/execution/modes/webhook.ts` — same
- Verify `TradeProposal`, `EntryGateCandidate`, and the execution call site all use the same field name

**Tests required:**
- Originator proposal with `confidence: 0.72` → prediction row has `model_probability = 0.72`
- Quant path candidate (no `confidence`) → prediction row created, `model_probability` null, row excluded from `learning_examples`

**Done when:** A new prediction row created via the originator path has a non-null `model_probability` distinct from `market_probability`.

### Task 0c — `feat/v1.99-reactive-trigger-fix`
**Debug and fix ta_alert and event trigger paths**

**Branch base:** `release/v1.99`

**Context:** All 59 originator proposals are `trigger_reason = 'cadence'`. `ta_alert` and `event` have never fired in production. The reactive paths exist in code but are dead. This means the learning dataset accumulates only cadence-triggered decisions — systematically slower to build and potentially different in quality from reactive ones. It also means TA alert and event signals contribute zero data to calibration.

**Changes:**
- Trace `ta_alert` trigger path in `src/core/autonomous.ts` (or wherever originator is called): find why `triggerReason: 'ta_alert'` never fires — likely the alert condition is never met or the code path that sets it is not reached
- Same for `event` trigger — find the dead code path
- Fix so that when a TA alert fires or a relevant event is detected, the originator is called with the correct `triggerReason`
- Add logging at the trigger decision point so the reason is visible in production logs

**Tests required:**
- TA alert condition met → originator called with `triggerReason: 'ta_alert'`
- Event detected → originator called with `triggerReason: 'event'`
- No alert/event → originator called with `triggerReason: 'cadence'`

**Done when:** At least one `ta_alert` or `event` proposal appears in `llm_trade_proposals` in production within one week of deployment.

## Phase 3 — Trade Instrumentation (discovered 2026-04-03)

**Context:** Production DB audit found that `mfeProxy`, `actualPnlPct`, `capturedR`, and `signalClass` are in the `perp_trade_journal` schema but never written. 276 journal entries, 0 with MFE/PnL data. 81 resolved trades, all with `signalClass = null`. The exit quality question — are exits cutting winners short? — cannot be answered from current data. This is the same instrumentation integrity problem as the `predicted_probability` contamination, applied to trade-level attribution.

**Gate:** Start after Phase 1 is merged. No sample-count gate (instrumentation writes immediately).

### Task 5 — `feat/v1.99-trade-pnl-attribution`
**Write MFE/MAE watermarks and actualPnlPct on position close**

**Branch base:** `release/v1.99`

**Changes:**
- `src/core/position_heartbeat.ts` — on each heartbeat tick, update running high/low watermarks in `position_exit_policy` or a new `trade_watermarks` table (`mfe_pct`, `mae_pct` as max adverse/favorable seen since open)
- `src/core/tool-executor.ts` (or wherever position closes are written) — on close, compute `actualPnlPct = (exit - entry) / entry * side_sign`, `capturedR = actualPnlPct / maeProxy` if stop was set, and write them into the journal entry for that trade
- `src/memory/perp_trade_journal.ts` — confirm write sites for `mfeProxy`, `maeProxy`, `actualPnlPct`, `capturedR` are called on close

**Tests required:**
- Open + tick up 3% + close: journal entry has `mfeProxy ≈ 0.03`, `actualPnlPct ≈ 0.03`
- Open + tick up 5% + close at 2%: `mfeProxy ≈ 0.05`, `actualPnlPct ≈ 0.02`, capture rate ≈ 40%
- Open + tick down (adverse) + close at loss: `maeProxy` reflects max drawdown seen

**Done when:** New closed trades in production have non-null `mfeProxy`, `maeProxy`, `actualPnlPct`. Capture rate query returns real data.
