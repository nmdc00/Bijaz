# v1.99 Sprint Plan
## Performance & Learning Integrity Layer (PLIL)

**Release branch:** `release/v1.99`
**Status:** Planning
**Date:** 2026-04-02
**Depends on:** v1.97 shipped and stable in production

---

## Sprint Goal

Make every prediction row carry a clean, auditable separation between what Thufir believed and what the market implied. Ensure outcome labels are trustworthy. Close the loop from clean data to gate enforcement.

This sprint has two phases. Phase 2 does not start until Phase 1 has produced 50+ `final`-resolution predictions in the production DB.

---

## Phase 1 — Data Integrity (start immediately)

### Task 1 — `feat/v1.99-schema-probability-split`
**Schema migration + write site fixes**

**Branch base:** `release/v1.99`

**Changes:**
- `src/memory/schema.sql` — add `model_probability REAL`, `market_probability REAL`, `outcome_basis TEXT DEFAULT 'legacy'`, index, and `learning_examples` view (see TDD §1)
- `src/memory/db.ts` — guarded migration for new columns (use `tableHasColumn` pattern, wrap each ALTER in try/catch)
- `src/memory/predictions.ts` — add `modelProbability` and `marketProbability` to `PredictionInput`, `PredictionRecord`, INSERT params, SELECT mapping
- `src/execution/modes/paper.ts:97` — pass `modelProbability: decision.probability`, `marketProbability: market.prices?.[decision.outcome]`
- `src/execution/modes/webhook.ts:24` — same
- Confirm `decision.probability` is present on the candidate object at execution time; if not, add it to the originator output type and thread it through

**Tests required:**
- `createPrediction` with both fields — correct DB write
- `createPrediction` without `modelProbability` — row created but absent from `learning_examples` view
- Paper trade end-to-end: candidate with `probability: 0.72`, market price `0.61` → prediction row has `model_probability = 0.72`, `market_probability = 0.61`

**Done when:** New prediction rows in production have non-null, distinct `model_probability` and `market_probability`. `outcome_basis` defaults to `'legacy'` for all existing rows, `'estimated'` or `'final'` for all new rows after resolver fix.

---

### Task 2 — `feat/v1.99-resolver-outcome-basis`
**Resolver produces trustworthy outcome labels**

**Branch base:** `release/v1.99`
**Depends on:** Task 1 merged (needs new column)

**Changes:**
- `src/memory/calibration.ts` — add `outcomeBasis?: 'final' | 'estimated'` to `recordOutcome` params; write it in the UPDATE statement; default to `'estimated'` if not passed
- `src/core/resolver.ts` — pass `outcomeBasis: 'final'` when market is confirmed resolved via API; pass `outcomeBasis: 'estimated'` when falling back to snapshot threshold (the `basis: 'snapshot_threshold'` branch)

**Tests required:**
- Resolver with confirmed market resolution → `outcome_basis = 'final'`
- Resolver with snapshot fallback → `outcome_basis = 'estimated'`
- `'estimated'` row is absent from `learning_examples` view
- `'final'` row appears in `learning_examples` view

**Done when:** The resolver never promotes a snapshot-threshold resolution to `'final'`. The `learning_examples` view is clean.

## Gate check (between phases)

Before starting Phase 2: query production DB.

```sql
SELECT COUNT(*) FROM learning_examples;
```

If result < 50, wait. Phase 2 metrics are meaningless below this threshold. Monitor daily.

## Phase 2 — Metrics + Gate Wiring (start when 50+ final predictions exist)

### Task 3 — `feat/v1.99-learning-metrics`
**Rolling window metrics engine**

**Branch base:** `release/v1.99`
**Depends on:** Tasks 1 + 2 merged; 50+ rows in `learning_examples`

**Changes:**
- `src/memory/learning_metrics.ts` — new file; `computeRollingWindowMetrics(domain?)`, `computeDomainWindowMetrics()` (see TDD §4)
- `src/gateway/dashboard_api.ts` — add `buildPredictionAccuracySection()`, add `predictionAccuracy` to `buildDashboardApiPayload()` sections
- `dashboard/src/types.ts` — add `predictionAccuracy` to `DashboardPayload.sections`
- `dashboard/src/App.tsx` — extend the existing `performance` tab to render rolling window cards (window size, sample count, accuracy, Brier model, Brier market, Brier delta, avg edge, PnL); show "-" for any metric where `sampleCount < 20`

**Tests required:**
- `computeRollingWindowMetrics()` with 15 seeded rows — all metrics null
- `computeRollingWindowMetrics()` with 25 seeded rows — window-10 and window-20 populated, others null
- Brier delta positive when model outperforms market baseline
- `buildPredictionAccuracySection()` returns correct shape with empty DB (no throw)

**Done when:** The dashboard performance tab shows rolling window cards. A row with `totalFinalPredictions = 0` renders cleanly without errors.

---

### Task 4 — `feat/v1.99-gate-calibration-enforcement`
**Entry gate enforces calibration-aware domain blocks**

**Branch base:** `release/v1.99`
**Depends on:** Task 3 merged

**Changes:**
- `src/core/llm_entry_gate.ts` — add `calibrationBlock(domain)` function; call it at the top of `evaluate()` before the LLM call; apply `sizeMultiplier` to suggested risk % when gate approves; log reason in gate log entry (see TDD §5)

**Tests required:**
- `calibrationBlock` with < 20 samples → `blocked: false`, `sizeMultiplier: 1`
- `calibrationBlock` with 50+ samples, Brier delta < 0 → `blocked: true`, reason `'domain_calibration_below_market'`
- `calibrationBlock` with 20+ samples, Brier delta < -0.05 → `sizeMultiplier: 0.5`
- `calibrationBlock` with 50+ samples, Brier delta > 0 → `blocked: false`, `sizeMultiplier: 1`
- Integration: gate rejects entry for blocked domain, logs reason

**Done when:** A domain with confirmed negative Brier delta over 50 samples is blocked at the gate with a logged reason. Telegram notification fires (reuse existing notify path).

## What is NOT in this sprint

- Calibration scalar applied at inference time (needs ~200 per-domain samples to be reliable)
- Automated drift detection (manual inspection sufficient; add in v1.100 if justified)
- New dashboard tab (rolling windows go in the existing performance tab)
- Backfilling legacy rows
