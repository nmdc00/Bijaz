# v1.98 Vision Document
## LLM-Primary Trading: From Gatekeeper to Originator

**Status:** Planned — begin after v1.97 ships and 30–50 clean trades are recorded
**Date:** 2026-03-19

---

## The Core Shift

v1.97 makes the LLM a **gatekeeper**: the quant pipeline proposes, the LLM approves or rejects.

v1.98 makes the LLM a **originator**: the LLM proposes trade ideas based on narrative and market context; the quant validates sizing, risk, and regime suitability only.

These are fundamentally different systems. The distinction matters:

| | v1.97 (gatekeeper) | v1.98 (originator) |
|---|---|---|
| Who generates the idea | Quant (signals → hypothesis) | LLM (narrative → thesis) |
| Who decides direction | Quant (signal class, regime) | LLM |
| Who manages sizing | Quant (Kelly, session weight) | Quant (unchanged) |
| Who manages exit | LLM exit consultant | LLM exit consultant (unchanged) |
| Default posture | No trade unless LLM approves | No trade unless LLM proposes |

---

## Why This Is the Right Sequence

### The quant pipeline's world model is broken

The system ran ~483 trades. `momentum_breakout` (151 samples) and `mean_reversion` (89 samples) — the two primary signal archetypes — produced 0% win rates. This is not a calibration problem. It means the quant's classification of what kind of trade it's generating does not correspond to how the market actually moves. You cannot fix a broken world model by tuning thresholds.

### Crypto is narrative-driven

Price in crypto moves on stories: ETF approvals, protocol exploits, macro fear, regulatory news, influencer narratives, funding rate extremes. These are exactly the inputs an LLM natively processes. The quant has no mechanism to reason about them.

### The LLM's directional judgment has been correct

In observed trades, the LLM gave coherent directional analysis for ZEC (short) and CL (short) that was correct. The failures were in position management — not in direction. That is the signal: the LLM's narrative judgment is already better than the quant's signal classification.

### LLMs overtrade without explicit constraint

LLMs are trained to be helpful and coherent, not to be uncertain. In trading, the highest-value skill is knowing when not to trade. Without explicit design, an LLM will find a reason to act on almost anything. This is the primary risk of LLM-primary trading and must be addressed in prompt architecture.

## v1.98 Architecture

### New path: LLM Trade Origination

Alongside the existing discovery pipeline, a new path:

```
LlmTradeOriginator
  ├── Input: current book (PositionBook)
  │          market context (gatherMarketContext)
  │          recent news / event feed
  │          signal performance summary (from Phase 2 feedback loop)
  │          current session / regime context
  │
  ├── System prompt: "You are Thufir. Your default is NO TRADE.
  │                   Only propose a trade if you have a strong narrative
  │                   thesis AND the setup is clean. Maximum 1 proposal
  │                   per scan cycle. If nothing is compelling, return null."
  │
  ├── Output: TradeProposal | null
  │           { symbol, side, thesisText, invalidationCondition,
  │             suggestedTtlMinutes, confidence }
  │
  └── If TradeProposal returned:
        → Quant validates: sizing (Kelly), risk limits, leverage cap
        → PositionBook.hasConflict() check
        → executor.execute() if validated
        → exit policy written with LLM's invalidationCondition + ttl
```

### Existing quant path: demoted but not removed

The discovery pipeline continues to run. Its candidates go through the v1.97 entry gate (LLM approval still required). But the LLM origination path gets priority — if the LLM has originated a position on a symbol, the quant cannot also open one.

### Prompt design principles (critical)

1. **Hard single-proposal constraint:** The prompt must cap the LLM at 1 proposal per scan. Without this, it will try to fill the whole daily budget in one shot.
2. **Explicit null is valid:** The JSON schema must treat `null` as a first-class return value and the prompt must frame it as the expected outcome, not a failure.
3. **Thesis quality bar:** The proposal must include `invalidationCondition` (what would make this thesis wrong) and `suggestedTtlMinutes`. No thesis = no trade. This prevents vague approvals.
4. **Feed the track record:** The prompt receives the LLM's own gate log performance summary. It should know what setups it has approved and whether they won. This is the core of the feedback loop.

## Prerequisites Before v1.98 Can Begin

1. **v1.97 shipped and stable** — LLM entry gate and exit consultant running in production
2. **30–50 clean trades recorded** — enough data for the LLM's own track record to be meaningful
3. **Phase 2 P2-T1 complete** — `unknown` signal/regime diagnosis fixed; classification is reliable
4. **Phase 2 P2-T2 complete** — nightly scoring job running; `llm_gate_performance` table populated
5. **Historical data cleared** — contaminated dataset purged before v1.98 data collection begins

---

## What v1.98 Does NOT Change

- The quant pipeline is not removed. It becomes a secondary origination path.
- Sizing and risk management remain fully deterministic (quant-owned).
- Exit management remains with the LLM exit consultant (v1.97).
- The fallback chain (main LLM → trivial → reject) is unchanged.

---

## Success Criteria for v1.98

- LLM-originated trades account for >50% of executed trades within 2 weeks of shipping
- Win rate on LLM-originated trades measurably higher than quant-only baseline (>25%)
- Null-proposal rate (LLM decides not to trade) > 60% of scan cycles — confirms the "default no trade" discipline is holding
- No simultaneous long+short on same symbol (inherited from v1.97)

## Open Questions for v1.98 Design Phase

1. **How frequently should `LlmTradeOriginator` run?** Same cadence as discovery scan (15 min), or event-triggered (on new market context / news event)?
2. **Should the LLM see the quant's candidates before proposing?** Option: show it the quant's top 3 ranked expressions as "here is what the signal pipeline is seeing" — the LLM can either endorse one, override it, or ignore all of them and propose something different.
3. **How do we prevent the LLM from always proposing the same few symbols?** May need a diversity constraint or cooldown per symbol.
4. **Live mode:** LLM-originated trades with real money require a confidence threshold gate. Define it before v1.98 goes live.
