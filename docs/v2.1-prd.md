# v2.1 Product Requirements Document
## Extractable Learning Substrate

**Release branch:** `release-v2.1`  
**Status:** Draft  
**Date:** 2026-05-05  
**Depends on:** `release-v2.00` baseline, PLIL phase 2 wiring live

---

## Problem Statement

Thufir can now measure some of its forecasting performance, but its learning loop is not yet extractable as a reusable framework.

The current code mixes together:

- comparable forecast learning
- perp execution learning
- journal/audit artifacts
- dashboard/reporting surfaces
- policy updates

This creates three concrete failures:

1. **Comparator contamination in perps.**  
   Autonomous perp predictions are still written with `marketProbability = 0.5` and `learningComparable = true`, which makes them look methodology-valid even when no real baseline exists.

2. **Fragmented learning records.**  
   Relevant evidence is split across `predictions`, `learning_events`, `decision_artifacts`, `perp_trade_journal`, and close-resolution metadata, without one canonical cross-domain learning case abstraction.

3. **Sparse, brittle execution learning.**  
   The adaptive perp learner relies on exact segment matches and partial journal completeness, which wastes production data and makes generalization weak.

The goal of `v2.1` is not "add alpha." The goal is:

> **Make Thufir's improvement loop explicit, reusable, and honest about what type of learning is happening.**

## Goal

Create an extractable learning substrate with two explicit tracks:

1. **Comparable Forecast Learning**
   - requires belief, baseline, and final outcome
   - drives calibration metrics and forecast-policy gating

2. **Execution Quality Learning**
   - requires action, context, realized outcome, and quality decomposition
   - drives sizing, timing, exit, and archetype-level policy adaptation

Thufir should become a producer of learning cases and a consumer of policy outputs, rather than a bundle of ad hoc learning paths.

---

## Non-Goals

- New trading strategies
- A perp comparator design that pretends to be solved when it is not
- Full backfill of legacy evidence
- Deployment automation changes
- Replacing the existing dashboard information architecture

---

## Requirements

### 1. Canonical Learning Case

Introduce a reusable learning-case abstraction that can represent both:

- comparable forecast cases
- execution-quality cases

Each case must be able to store:

- `case_type`
- `domain`
- `entity`
- `belief`
- `baseline`
- `comparable`
- `context`
- `action`
- `outcome`
- `quality_scores`
- `policy_inputs`
- `source links` back to prediction/trade/artifact rows

### 2. Comparator Hygiene

Perp rows must not enter comparable-learning metrics unless a real comparator exists.

Rules:

- no synthetic `0.5` baseline for comparable learning
- no `learningComparable = true` for perps by default
- comparable learning must remain honest and queryable

### 3. Execution Learning Track

Closed perp trades must produce normalized execution-learning evidence containing:

- thesis correctness
- direction quality
- timing quality
- sizing quality
- exit quality
- `capturedR`
- `leftOnTableR`
- MAE/MFE proxies
- realized PnL / fees
- regime/vol/liquidity context

### 4. Hierarchical Learning

Execution learning must no longer rely only on exact segment matches.

The learner must support hierarchical backoff such as:

- exact segment
- partial segment
- signal-class level
- global prior

### 5. Explicit Policy Output

Learning modules must emit policy guidance in an explicit, reusable form:

- `block`
- `resize`
- `bias`
- `suppress`
- `prior_adjustment`

The domain runtime may apply these outputs differently, but the learning layer must produce them deterministically.

### 6. Auditability

The system must provide an auditable explanation of:

- why a row is comparable or not
- which learning track consumed it
- which policy output changed because of it

## Architecture

### Before

```text
domain-specific write paths
  -> predictions / journals / artifacts / learning_events
  -> mixed semantics
  -> mixed comparable + non-comparable cases
  -> partial policy updates
```

### After

```text
runtime decision
  -> canonical learning case writer
    -> comparable forecast case OR execution-quality case
      -> learner
        -> policy output
          -> gate / sizing / prioritization / suppression
```

## Sprint Scope

### Task 1 — `feat/v2.1-learning-case-foundation`

Create the canonical learning-case schema, migration, typed memory access layer, and audit views.

### Task 2 — `feat/v2.1-comparator-hygiene`

Remove fake comparable semantics from perps and refactor prediction outcome recording so PLIL-native fields are the source of truth.

### Task 3 — `feat/v2.1-execution-learning-engine`

Write normalized execution-quality cases from perp closes and implement hierarchical adaptive learning on top of them.

### Task 4 — `feat/v2.1-policy-audit-surface`

Expose learning-track counts, comparable exclusions, and policy outputs in the dashboard/API so the framework is explainable.

---

## Success Criteria

`v2.1` is complete when:

- perps are excluded from comparable learning unless a real comparator is present
- canonical learning cases exist for both comparable forecasts and execution-quality outcomes
- closed perp trades generate normalized execution-learning records
- adaptive execution learning no longer requires exact segment-only matches
- policy outputs are explicit and traceable
- dashboard/API can answer:
  - how many comparable cases exist?
  - how many execution cases exist?
  - how many rows were excluded and why?
  - what policy changes are currently active?

---

## Risks

- Migration complexity across overlapping legacy tables
- Dashboard overreach if audit surfaces are mixed into user-facing trading summaries
- False confidence if hierarchical backoff is too aggressive on small samples
- Release coordination risk because `src/core/autonomous.ts` and `src/core/tool-executor.ts` are hot files

---

## External Framing

The key extractable claim after `v2.1` should be:

> Thufir does not merely store outcomes. It converts runtime decisions into canonical learning cases, separates forecast calibration from execution learning, and turns those cases into explicit policy updates.
