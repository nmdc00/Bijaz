# v1.97 Sprint Plan
## LLM-Coherent Execution: Shared Book, Entry Gate, Adaptive Exit

**Release branch:** `release/v1.97`
**Status:** Planning
**Date:** 2026-03-19

---

## Sprint Goal

Make the LLM a first-class participant in trade entry and exit decisions. Eliminate the possibility of self-cancelling positions. Give every open position a live LLM guardian.

---

## Task Breakdown

### Task 1 — `feat/v1.97-position-book`
**PositionBook singleton — shared live view of open positions**

- `src/core/position_book.ts` — `PositionBook` class (singleton, in-memory, DB-backed)
- `refresh()` reads from `listOpenPositionsFromTrades` + `position_exit_policy`
- `hasConflict(symbol, side)` — detects opposite-side conflicts
- `getAll()` returns enriched `BookEntry[]`
- Wire `PositionBook.refresh()` into `AutonomousManager` scan start
- Wire `PositionBook.refresh()` into `PositionHeartbeatService` tick start
- **Tests:** unit tests for `hasConflict`, `refresh`, `getAll`; integration test confirming book is populated after a paper trade

**Foundational — must land first.**

---

### Task 2 — `feat/v1.97-llm-entry-gate`
**LLM entry gate — review and approve/reject/resize every candidate before execution**

- `src/core/llm_entry_gate.ts` — `LlmEntryGate` class
- `evaluate(candidate, markPrice)` → `EntryGateDecision`
- Conflict fast-path: if `PositionBook.hasConflict`, return `reject` without LLM call
- Main LLM call with structured JSON output (5s timeout)
- Fallback to trivial client on timeout/error + `notify()` to user
- Default to `reject` if both fail
- Integrate into `AutonomousManager` before `executor.execute()`
- DB: `llm_entry_gate_log` table + insert helper
- **Tests:** mock LLM → approve/reject/resize paths; conflict fast-path; fallback fires notify; DB write verified

**Depends on Task 1.**

---

### Task 3 — `feat/v1.97-llm-exit-consultant`
**LLM exit consultant — event-driven position re-evaluation during hold**

- `src/core/llm_exit_consultant.ts` — `LlmExitConsultant` class
- `consult(position, currentPrice, freshContext)` → `ExitConsultDecision`
- Trigger conditions: 20-min cadence, ROE milestones (3/7/15%), approach TTL (15 min)
- Actions: `hold`, `reduce`, `close`, `extend_ttl`, `update_invalidation`
- Main LLM with fallback to trivial + notify; default `hold` on both fail
- Integrate into `PositionHeartbeatService.tickOnce()`
- DB: `llm_exit_consult_log` table + insert helper
- **Tests:** trigger condition logic (time, ROE, TTL); each action path executes correct downstream call; fallback path; DB write verified

**Depends on Task 1.**

---

### Task 4 — `feat/v1.97-gateway-llm-wiring`
**Wire main + trivial LLM clients into heartbeat service via gateway**

- Pass `primaryAgent.llm` and `primaryAgent.infoLlm` into `PositionHeartbeatService` constructor
- Update `PositionHeartbeatService` constructor signature accordingly
- Add config keys: `positionHeartbeat.llmExitConsult.*` and `autonomy.llmEntryGate.*`
- **Tests:** gateway integration test confirming `PositionHeartbeatService` receives both clients

**Depends on Tasks 2 and 3.**

---

### Task 5 — `feat/v1.97-integration-acceptance`
**Integration and acceptance tests for the full coherence pipeline**

- End-to-end test: autonomous scan with a conflicting position in DB → entry gate rejects, no order placed
- End-to-end test: position open 20+ min → exit consultant fires, DB log entry written
- End-to-end test: main LLM unavailable → fallback fires, notify called
- Regression: existing heartbeat trigger paths (time stop, invalidation, ROE close) still work without LLM in the way
- `pnpm test && pnpm typecheck` fully green

**Depends on Tasks 1–4.**

## Risks

| Risk | Mitigation |
|------|-----------|
| LLM latency blocks entry in fast-moving markets | 5s timeout + immediate fallback; miss a trade rather than freeze |
| LLM JSON output malformed | Strict Zod parse; any parse error = reject/hold (safe default) |
| Exit consultant fires on every tick = token cost | Event-driven trigger conditions; max ~3 consults per position per hour |
| Fallback trivial client gives poor exit decisions | Notify user; trivial defaults conservative (hold/reject) |
| Phase 2 feedback loop creates circular bias | Addressed in phase 2 design — out of scope this sprint |

## Phase 2 Sprint (planned — after v1.97 ships)

**Precondition:** Clear all historical performance data before Phase 2 begins. The ~483 trade dataset is contaminated (self-cancelling positions, broken paper equity state, arbitrary heartbeat closes). Starting clean is required for the feedback loop to produce valid signal.

### Phase 2 Tasks

**P2-T1 — Diagnose and fix `unknown` signal class / regime**
- Trace `classifySignalClass()` and `classifyMarketRegime()` in `autonomy_policy.ts`
- Identify when/why classification falls through to `'unknown'` (likely: missing context pack data, incomplete snapshot at classification time)
- Add logging at classification boundary to capture raw inputs on `unknown` returns
- Fix root cause; success = `unknown` < 10% of trades in clean data

**P2-T2 — Nightly LLM decision scoring job**
- For each `llm_entry_gate_log` row with a resolved trade outcome: score the decision (was approve/reject correct?)
- Write `signal_class_score`, `regime_score`, `session_score` per combination into a new `llm_gate_performance` table
- Surface via `summarizeSignalPerformance()` so it's available to future entry gate prompts

**P2-T3 — Signal performance context in entry gate prompt**
- Populate `signalPerformanceSummary` field in `LlmEntryGate.evaluate()` prompt with live data from `llm_gate_performance`
- The LLM now reads its own track record as a prior: "momentum_breakout: 40 decisions, 38 wrong — need strong narrative to override"
- This closes the feedback loop: LLM decisions → outcome data → LLM prior → better decisions

**P2-T4 — Multi-position portfolio reasoning (stretch)**
- LLM entry gate receives the full book, not just the candidate trade
- Can reason about correlation, concentration, and overall posture before approving
