# v1.97 Technical Design Document
## LLM-Coherent Execution: Shared Book, Entry Gate, Adaptive Exit

**Release branch:** `release/v1.97`
**Status:** Planning
**Date:** 2026-03-19

---

## Architecture Overview

Three new subsystems are introduced. All interact with existing code through narrow, well-defined interfaces to minimise blast radius.

```
Discovery Pipeline          LLM Entry Gate              Execution
──────────────────          ──────────────              ─────────
runDiscovery()         →    LlmEntryGate.evaluate()  →  executor.execute()
  expressions[]             (main LLM / trivial fallback)
                                  │
                                  ▼
                         EntryGateDecision
                         { verdict: APPROVE|REJECT|RESIZE
                           reasoning: string
                           adjustedSize?: number }

Position Heartbeat          LLM Exit Consultant
──────────────────          ───────────────────
tick() [every 60s]     →    LlmExitConsultant.consult()   →   updateExitPolicy() / close()
  per position               (main LLM / trivial fallback)      (existing heartbeat executor)
  [event-driven trigger]          │
                                  ▼
                         ExitConsultDecision
                         { action: HOLD|REDUCE|CLOSE|EXTEND_TTL|UPDATE_INVALIDATION
                           reasoning: string
                           newTimeStopAtMs?: number
                           newInvalidationPrice?: number }

Shared Book State
─────────────────
PositionBook (singleton, in-memory + DB-backed)
  - open positions (from DB, refreshed each heartbeat tick)
  - per-position: entry reasoning, thesis, last LLM consult timestamp
  - read by: LlmEntryGate, LlmExitConsultant, ConversationHandler
  - written by: AutonomousManager (on open), heartbeat (on close), exit consultant (on update)
```

## Component Designs

### 1. PositionBook (`src/core/position_book.ts`)

A lightweight in-process singleton that holds the current view of all open positions enriched with thesis data. Both the LLM and the autonomous system read from it.

**State per position:**
```typescript
interface BookEntry {
  symbol: string;
  side: 'long' | 'short';
  size: number;
  entryPrice: number;
  entryReasoningText: string;   // from TradeDecision.reasoning (already stored)
  thesisExpiresAtMs: number;    // from exit policy
  lastConsultAtMs: number | null;
  lastConsultDecision: ExitConsultDecision | null;
}
```

**Key methods:**
- `PositionBook.getInstance(): PositionBook`
- `refresh(): Promise<void>` — re-reads from DB (`listOpenPositionsFromTrades`)
- `get(symbol): BookEntry | undefined`
- `getAll(): BookEntry[]`
- `hasConflict(symbol, side): boolean` — returns true if opposite side open

**Notes:**
- Backed by the existing `perp_trades` and `position_exit_policy` tables — no new schema needed for phase 1.
- Refreshed at the start of each heartbeat tick and at the start of each autonomous scan cycle.
- The `ConversationHandler` can expose `PositionBook.getAll()` to the LLM as ambient context via a tool (`get_open_positions` already exists — wire it to this).

### 2. LlmEntryGate (`src/core/llm_entry_gate.ts`)

Called by `AutonomousManager` for each candidate expression before `executor.execute()`.

**Interface:**
```typescript
interface EntryGateDecision {
  verdict: 'approve' | 'reject' | 'resize';
  reasoning: string;
  adjustedSizeUsd?: number;   // only if verdict === 'resize'
}

class LlmEntryGate {
  constructor(
    private mainLlm: LlmClient,
    private fallbackLlm: LlmClient,          // trivial
    private notify: (msg: string) => Promise<void>,
    private book: PositionBook,
    private config: ThufirConfig,
  ) {}

  async evaluate(candidate: ExpressionCandidate, markPrice: number): Promise<EntryGateDecision>
}
```

**Prompt inputs (assembled in `evaluate`):**
1. Current book (symbol, side, size, unrealised PnL, time held, thesis summary) — from `PositionBook.getAll()`
2. Candidate trade (symbol, side, notional, leverage, edge, confidence, entry reasoning from expression)
3. Conflict flag: is there an opposite-side position open on this symbol?
4. Instruction: return structured JSON with `verdict`, `reasoning`, optional `adjustedSizeUsd`

**Conflict fast-path:** If `PositionBook.hasConflict(symbol, side)` is true, skip LLM and return `reject` immediately with reasoning "opposite side already open". This is deterministic and free.

**Fallback:** If main LLM call throws or times out (5s budget):
- Retry once with fallback trivial client
- Fire `notify()` with "⚠️ Entry gate: using fallback LLM (quality may be lower)"
- If both fail, default to `reject` (safe default — miss a trade rather than execute without review)

**Persistence:** Each `EntryGateDecision` is written to a new `llm_entry_gate_log` table (see schema section).

### 3. LlmExitConsultant (`src/core/llm_exit_consultant.ts`)

Called by `PositionHeartbeatService` for each open position on a trigger-based cadence.

**Interface:**
```typescript
interface ExitConsultDecision {
  action: 'hold' | 'reduce' | 'close' | 'extend_ttl' | 'update_invalidation';
  reasoning: string;
  newTimeStopAtMs?: number;
  newInvalidationPrice?: number;
  reduceToFraction?: number;   // e.g. 0.5 for 50% reduce
}

class LlmExitConsultant {
  constructor(
    private mainLlm: LlmClient,
    private fallbackLlm: LlmClient,
    private notify: (msg: string) => Promise<void>,
    private config: ThufirConfig,
  ) {}

  async consult(position: BookEntry, currentPrice: number, freshContext: string): Promise<ExitConsultDecision>
}
```

**Prompt inputs:**
1. Original entry reasoning (from `BookEntry.entryReasoningText`)
2. Current position state (ROE, time held, current price vs entry, time remaining on thesis)
3. Fresh market context string — pulled from the existing `gatherMarketContext` utility (already available from v1.95, limit 500 chars to keep tokens tight)
4. Instruction: structured JSON response with `action` + `reasoning` + optional update fields

**Trigger conditions (event-driven, not every tick):**
- Time-based: first consult at 20 min after entry, then every 20 min while position is open
- ROE milestone: consult when |ROE| crosses 3%, 7%, 15% for the first time (thresholds configurable)
- Approach TTL: consult when `timeStopAtMs - now < 15 min` (give the LLM a chance to extend or commit to close)

**Fallback:** Same pattern as entry gate — retry trivial client, notify user, default to `hold` if both fail (safer than defaulting to close).

**Persistence:** Each decision written to `llm_exit_consult_log` table.

### 5. Integration Points

**`AutonomousManager` (`src/core/autonomous.ts`):**
- Instantiate `LlmEntryGate` with main + trivial clients and book reference
- After candidate sizing, before `executor.execute()`: call `gate.evaluate(candidate, markPrice)`
- On `reject`: skip, log, continue to next candidate
- On `resize`: replace `probeUsd` with `adjustedSizeUsd`, continue
- On `approve`: proceed as today
- At scan start: call `PositionBook.getInstance().refresh()`

**`PositionHeartbeatService` (`src/core/position_heartbeat.ts`):**
- Instantiate `LlmExitConsultant` in constructor (receives main + trivial + notify)
- In `tickOnce()`, after loading positions: for each position check if consult trigger conditions are met
- If triggered: call `consultant.consult(bookEntry, currentPrice, freshContext)`
- On `close`: fire existing close path
- On `reduce`: fire existing reduce path
- On `extend_ttl` / `update_invalidation`: call `upsertPositionExitPolicy` with new values
- On `hold`: just log

**`gateway/index.ts`:**
- Pass main LLM client into `PositionHeartbeatService` constructor (currently it only gets `toolContext`)
- Pass trivial LLM client as fallback

## Phase 2: LLM Teaches Quant (Feedback Loop — not in this sprint)

### Data Reset Precondition

Before Phase 2 begins, all historical performance data must be cleared. The existing dataset (~483 trades) is contaminated: self-cancelling positions (simultaneous long+short same symbol), trades closed by arbitrary time stops rather than logical exits, and a period of broken paper equity (-$1.4k state). Measuring against this data would produce false priors. Phase 2 starts from a clean slate after v1.97 ships.

Tables to truncate before Phase 2: `perp_trades`, `perp_trade_journals`, `llm_entry_gate_log`, `llm_exit_consult_log`, `position_exit_policy`, `autonomy_policy_state`, signal performance tables.

---

### The Feedback Loop Model

```
Quant generates candidates
        │
        ▼
LlmEntryGate.evaluate(candidate + signalPerformanceSummary)
        │
   approve / reject / resize
        │
        ▼
Trade executes (or not)
        │
        ▼
Outcome recorded in perp_trades + llm_entry_gate_log
(signal_class, regime, session, verdict, outcome, PnL, R-multiple)
        │
        ▼
Nightly scoring job:
  for each entry gate decision:
    - was LLM right to approve? (outcome R > 0)
    - was LLM right to reject? (counterfactual: would the trade have lost?)
    → update signal_class_score, regime_score, session_score per combination
        │
        ▼
summarizeSignalPerformance() now reflects LLM-validated trades only
        │
        ▼
Next LLM entry gate call reads updated performance table as prior
→ sharper filtering → better trades → better data → loop tightens
```

The LLM is not hardcoding rules. It is reading its own track record and using it as evidence. Over time it effectively learns: "momentum_breakout in choppy regime has been wrong 40/40 times — I need a very strong narrative override to approve this." That policy lives in the data, not in the prompt.

### Phase 2 Task: Diagnose `unknown` Signal Class and Regime

The current dataset shows `unknown` as the dominant signal class (237/483 samples) and regime (283/483). This is a serious problem: if the quant pipeline cannot classify what kind of trade it's generating or what market context it's operating in, the feedback loop has nothing meaningful to learn from.

**Investigation tasks:**
1. Trace how `signal_class` is assigned in `autonomy_policy.ts` → `classifySignalClass()`. When does it fall through to `'unknown'`? What inputs cause that?
2. Trace how `regime` is assigned → `classifyMarketRegime()`. Same question.
3. Check whether classification happens before or after the execution path — if it's evaluated on incomplete data (e.g., missing OI or funding), it would default to `unknown`.
4. Add logging at the classification step to capture the raw inputs when `unknown` is returned.
5. Fix the root cause — likely: missing data guard, insufficient signal thresholds, or classifier never being called with a populated context pack.

**Success criterion:** In Phase 2 clean data, `unknown` should represent <10% of trades. The dominant categories should be real signal classes (`momentum_breakout`, `mean_reversion`, etc.) so the feedback loop has meaningful segments to score.

---

### Phase 2 Entry Gate Prompt Enhancement

The `LlmEntryGate.evaluate()` prompt (built in this sprint) includes a `signalPerformanceSummary` field that is empty or stub in v1.97. In Phase 2, this field is populated with the live signal performance table — win rate, expectancy, and sample count per signal class, regime, and session combination. The LLM reads it as a prior and uses it to calibrate its approval threshold. No prompt engineering changes needed — only the data input changes.
