# v1.97 Product Requirements Document
## LLM-Coherent Execution: Shared Book, Entry Gate, Adaptive Exit

**Release branch:** `release/v1.97`
**Status:** Planning
**Date:** 2026-03-19

---

## Problem Statement

The Thufir autonomous pipeline and the LLM agent operate in completely separate worlds. The bot executes trades without any LLM awareness; the LLM reasons about positions without any authority to influence them. This produces:

- **Contradictory book state:** The system opened simultaneous LONG + SHORT on PAXG and LONG HYPE while an existing SHORT HYPE was open — self-cancelling positions that only burn fees.
- **No mid-hold re-evaluation:** Exit policy (time stop + invalidation price) is written at entry and never updated. Markets change; the thesis can evolve; the system cannot adapt.
- **Wasted LLM capability:** Crypto moves are narrative- and news-driven. The LLM is well-suited to reason about this in real-time, but it never touches trade management.
- **No feedback loop:** The LLM's post-hoc reasoning never informs future quant decisions. Edge learned in conversation dies there.

---

## Goals

1. **Coherent book:** LLM and autonomous system share a real-time view of open positions and the current thesis for each.
2. **LLM entry gate:** Before any trade executes, the main LLM reviews the candidate against the current book and market narrative. Its decision gates or resizes the order.
3. **LLM-driven exit management:** The heartbeat consults the main LLM on a position-event or time cadence. LLM can update exit policy (extend, tighten, trigger early close).
4. **Graceful degradation:** If the main LLM is unavailable, fall back to the trivial client and notify the user that quality has dropped.
5. **Feedback loop (phase 2):** LLM entry/exit decisions are recorded as structured signal-quality data that the quant pipeline can consume in future cycles to improve hypothesis scoring.

---

## Non-Goals

- Replacing the deterministic discovery pipeline (signals → hypotheses → expressions). The quant engine remains the candidate generator.
- Real-time LLM price prediction. The LLM reasons about narrative, context, and coherence — not price levels.
- Multi-agent orchestration. All LLM calls use the existing `createAgenticExecutorClient` pattern.

---

## User Stories

**As the system operator:**
- I want the bot to never open a position that directly contradicts an existing one (same symbol, opposite side) without explicit LLM approval.
- I want the bot to re-evaluate open positions periodically and act on changed conditions — not just wait for a time stop or hard stop-loss.
- I want to be notified (Telegram) when the system falls back from the main LLM to the trivial client for a trade decision.

**As the LLM agent (Thufir):**
- I want to know what positions are open, why they were opened, and how they are performing — without being asked.
- I want my reasoning about a position to actually influence whether we hold, reduce, or close it.
- I want my assessment of a trade candidate to gate whether the execution engine fires.

---

## Success Criteria

- Zero simultaneous long+short on the same symbol unless explicitly approved.
- LLM exit consultation fires at least once per open position per 20 minutes.
- Fallback to trivial client produces a Telegram notification within 30 seconds.
- All LLM entry/exit decisions are persisted to DB for future feedback loop use.
- Full test suite passes (`pnpm test && pnpm typecheck`).
