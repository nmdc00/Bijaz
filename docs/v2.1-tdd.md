## 1. Data Model

### 1.1 `learning_cases` table

Add a new canonical table:

```sql
CREATE TABLE IF NOT EXISTS learning_cases (
  id TEXT PRIMARY KEY,
  case_type TEXT NOT NULL CHECK(case_type IN ('comparable_forecast', 'execution_quality')),
  domain TEXT NOT NULL,
  entity_type TEXT NOT NULL,
  entity_id TEXT NOT NULL,
  comparable INTEGER NOT NULL CHECK(comparable IN (0, 1)),
  comparator_kind TEXT,
  source_prediction_id TEXT,
  source_trade_id INTEGER,
  source_artifact_id INTEGER,
  belief_payload TEXT,
  baseline_payload TEXT,
  context_payload TEXT,
  action_payload TEXT,
  outcome_payload TEXT,
  quality_payload TEXT,
  policy_input_payload TEXT,
  exclusion_reason TEXT,
  created_at TEXT DEFAULT (datetime('now')),
  updated_at TEXT
);
```

Indexes:

```sql
CREATE INDEX IF NOT EXISTS idx_learning_cases_type ON learning_cases(case_type);
CREATE INDEX IF NOT EXISTS idx_learning_cases_domain ON learning_cases(domain);
CREATE INDEX IF NOT EXISTS idx_learning_cases_comparable ON learning_cases(comparable);
CREATE INDEX IF NOT EXISTS idx_learning_cases_prediction ON learning_cases(source_prediction_id);
CREATE INDEX IF NOT EXISTS idx_learning_cases_trade ON learning_cases(source_trade_id);
CREATE INDEX IF NOT EXISTS idx_learning_cases_entity ON learning_cases(entity_type, entity_id);
```

### 1.2 Views

Add two canonical views:

```sql
CREATE VIEW IF NOT EXISTS comparable_learning_cases AS
SELECT *
FROM learning_cases
WHERE case_type = 'comparable_forecast'
  AND comparable = 1;

CREATE VIEW IF NOT EXISTS execution_learning_cases AS
SELECT *
FROM learning_cases
WHERE case_type = 'execution_quality';
```

### 1.3 Compatibility

Keep `learning_examples` in place for `v2.1`, but make it a legacy-compatible surface rather than the only learning abstraction.

## 3. Comparator Hygiene

### 3.1 Autonomous perp prediction writes

In:

- `src/core/autonomous.ts`

Change perp prediction writes so that by default:

- `marketProbability` is `undefined` unless a real comparator exists
- `learningComparable = false`
- comparator metadata is written into the new learning-case surface as excluded / non-comparable

### 3.2 Prediction-market execution writes

In:

- `src/execution/modes/paper.ts`
- `src/execution/modes/webhook.ts`

Continue writing comparable prediction rows with real:

- `modelProbability`
- `marketProbability`

Also mirror them into `learning_cases` as `case_type = 'comparable_forecast'`.

### 3.3 Outcome recording

In:

- `src/memory/calibration.ts`

Refactor `recordOutcome()` so that:

- Brier/event semantics prefer `model_probability` and `market_probability`
- the associated comparable learning case is updated on outcome resolution
- the older `learning_events` path is preserved only as a secondary compatibility surface

## 4. Execution Learning Writer

### 4.1 Normalized execution cases

When a perp closes:

- `src/core/perp_lifecycle.ts`
- `src/core/tool-executor.ts`

Write an `execution_quality` learning case containing:

- symbol / domain / mode
- signal class
- regime / volatility / liquidity
- entry/exit prices
- path high/low if available
- realized PnL / fees
- thesis correctness
- `capturedR`
- `leftOnTableR`
- direction/timing/sizing/exit scores
- exit mode / invalidation context

### 4.2 Snapshot promotion

Use `perp_trade_snapshot` / journal payloads as evidence sources, but normalize them into `learning_cases` rather than forcing learners to query JSON artifacts directly.

## 5. Hierarchical Execution Learner

### 5.1 New module

Create:

- `src/core/execution_learning.ts`

Primary APIs:

- `computeExecutionSegmentSummary(filters)`
- `resolveHierarchicalExecutionEdge(input)`

### 5.2 Backoff order

For a given candidate:

1. exact match:
   - signalClass + marketRegime + volatilityBucket + liquidityBucket
2. partial:
   - signalClass + marketRegime + volatilityBucket
3. coarse:
   - signalClass + marketRegime
4. archetype:
   - signalClass
5. global prior

Output:

- `sampleCount`
- `expectancy`
- `confidenceWeight`
- `sourceLevel`

### 5.3 Replace or wrap existing adaptive edge

In:

- `src/core/adaptive_edge.ts`

Either:

- replace exact-match sampling with hierarchical backoff

or

- preserve the current API and delegate internally to the new hierarchical learner

The second option is preferred for minimal call-site churn.

## 8. Tests

### Task 1 foundation

- migration creates `learning_cases`
- `createLearningCase()` writes comparable and execution cases
- exclusion counts aggregate correctly

### Task 2 comparator hygiene

- autonomous perp prediction write defaults to `learningComparable = false`
- no fake `0.5` comparator enters comparable learning
- prediction-market comparable rows still populate correctly
- `recordOutcome()` updates the canonical comparable learning case

### Task 3 execution learner

- closed perp trade writes normalized execution-quality case
- hierarchical backoff falls from exact -> partial -> coarse -> prior
- adaptive edge resolves with correct source level and sample count

### Task 4 audit surfaces

- dashboard payload includes learning-audit section with empty DB
- production-like seeded DB reports comparable vs execution counts correctly
- excluded reasons render without throwing
