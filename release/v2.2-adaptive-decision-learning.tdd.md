# TDD: Adaptive Decision Learning — v2.2

## Scope

Deliver the first genuinely useful adaptive decision-learning layer for Thufir trading:

1. expand canonical dossier coverage,
2. add richer path and intervention capture,
3. strengthen structured review generation,
4. add explicit counterfactual objects and scoring,
5. add typed similarity retrieval at decision time,
6. add bounded policy adaptation from learned evidence,
7. expose enough observability to validate that the loop is changing future decisions.

## Architecture

v2.2 preserves a general learning architecture:

`DecisionCase -> Review -> CounterfactualSet -> Similarity Retrieval -> Policy Adaptation`

The implementation remains Thufir-specific through typed trading payloads.

Core layers:
- `dossier layer`: canonical lifecycle record
- `trajectory layer`: path / excursion / timing metrics
- `review layer`: structured evaluation of closed trades
- `counterfactual layer`: bounded alternative outcomes and intervention scoring
- `retrieval layer`: structured nearest-case lookup
- `policy layer`: bounded adaptation from aggregated evidence
- `enforcement layer`: deterministic application of retrieval and policy authority at the pre-execution seam

## Current Repo Findings

The current repository already exposes the exact runtime seam v2.2 must own:

- `src/core/autonomous.ts`
  - builds candidates
  - runs deterministic risk / wallet checks
  - calls `this.entryGate.evaluate(...)`
  - executes after approval
- `src/core/llm_entry_gate.ts`
  - accepts a structured `EntryGateCandidate`
  - returns `approve | reject | resize`
  - can suggest leverage
- `src/core/autonomy_policy.ts`
  - already supports deterministic downweight / pass / block style decisions
- `src/memory/learning_cases.ts`
  - still only models `comparable_forecast | execution_quality`
- `src/memory/learning_schema.ts`
  - still enforces the same narrow case-type schema

Therefore:
- the pre-execution control point already exists,
- the learning schema and runtime authority model do not yet match the desired v2.2 loop,
- v2.2 must upgrade both schema and runtime enforcement together.

## Runtime Behavior

### Decision-Time Enforcement Order

The existing `src/core/autonomous.ts` chain must be upgraded into this strict order:

1. candidate/proposal creation
2. deterministic hard risk checks
3. retrieval lookup
4. active policy adjustment lookup
5. adaptive decision enforcement
6. `LlmEntryGate.evaluate(...)` on the adjusted candidate
7. final deterministic execution checks
8. execution
9. dossier / trace persistence

This order is mandatory.

Retrieval and adaptive policy must act before the gate and before execution, not only after the fact.

### Open Path

On every executed open:
1. create or upsert dossier
2. persist request, context, gate, execution fields
3. derive and persist similarity features
4. persist retrieval payload used at decision time
5. persist active policy trace

Before step 1, the runtime must persist the enforcement result that produced the final candidate:
- original request
- retrieval modifiers
- policy modifiers
- final adjusted request
- final reject reason when blocked

### Close Path

On every full close:
1. update dossier path and close sections
2. compute structured review
3. compute counterfactual set
4. persist `execution_quality`
5. persist `thesis_quality`
6. optionally persist `intervention_quality`
7. optionally persist `regret_case`
8. refresh similarity features from the final review outcome

### Blocked / Rejected Path

For meaningful blocked decisions:
1. create lightweight non-executed dossier or decision artifact
2. persist gate verdict and requested action
3. mark for later regret analysis if a measurable opportunity unfolded

Blocked-path persistence is required because rejected candidates are a key source of regret learning and policy validation.

### Question 1: Can retrieval really alter live decisions?

Answer in code: yes, but through bounded deterministic modifiers.

Retrieval may alter:
- confidence
- size
- leverage cap
- confirmation requirement
- reject-band escalation

Retrieval may not:
- force execution
- override deterministic hard risk blocks
- mutate persistent policy directly

Integration points:
- proposal packaging in `src/core/autonomous.ts`
- gate candidate construction before `LlmEntryGate.evaluate(...)`
- dossier `retrieval` and `policy_trace` payloads

### Question 2: Can policy really resize or reject based on learned evidence?

Answer in code: yes, via deterministic enforcement before the gate and executor.

Policy may:
- resize notional
- cap leverage
- apply confidence haircuts
- require additional confirmation
- trigger cooldown
- reject when evidence-backed thresholds are crossed

Policy may not:
- operate without minimum evidence
- exceed bounded delta rules
- survive past expiry without revalidation
- hide its effects from the persisted decision trace

### Question 3: Can weak evidence be prevented from poisoning the loop?

Answer in code: yes, through evidence-quality gating.

Every retrieval/policy signal must carry:
- evidence count
- similarity strength
- review confidence
- source-quality score
- freshness
- contradiction score
- missing-data flags

Weak evidence behavior:
- may annotate the candidate
- may reduce confidence in learned guidance
- may not create hard reject rules by itself
- may not create large policy deltas

Required enforcement thresholds:
- minimum evidence count
- minimum retrieval confidence
- maximum allowed delta per policy update
- mandatory expiry / review horizon
- sparse-history penalty
- outlier protection

## Counterfactual Design

### Required Algorithms

The counterfactual engine must compute bounded alternatives:
- no-trade baseline: zero realized PnL and zero realized risk
- full-size baseline: scale outcome path by requested/approved ratio
- approved-size baseline: actual executed case
- delayed-entry baseline: reprice entry using nearest path-improvement heuristic when available
- invalidation-exit baseline: close at invalidation if invalidation is known
- TTL-exit baseline: close at the scheduled time-stop if available
- alternate-expression baseline: compare to alternate proxy/direct expression when linked candidate exists
- leverage-cap baseline: compare capped vs requested leverage
- opportunity-selection baseline: compare the chosen trade against the top rejected candidate when available

### Output Requirements

Each counterfactual must expose:
- assumptions
- confidence
- estimated value-add / value-destroy
- applicability flags

Weak or missing coverage must be explicit.

## Policy Adaptation Design

### Adaptation Triggers

Policy adaptation jobs run over dossier cohorts segmented by:
- signal class
- regime
- failure mode
- gate verdict
- entry extension bucket
- catalyst class
- execution-condition bucket
- opportunity-rank bucket
- session / transition bucket

### Adaptation Outputs

Policies can update:
- max size multipliers
- leverage caps
- entry stretch penalties
- required confirmation thresholds
- confidence haircuts
- cooldown windows
- reject thresholds for narrow, evidence-rich setup families only

### Guardrails

Every policy adjustment must require:
- minimum evidence count
- bounded delta
- confidence threshold
- freshness threshold
- contradiction tolerance
- expiry / review horizon
- operator visibility

No policy update may silently hard-disable a setup family from one or two trades.

## Regret Learning Design

Implement regret analysis for:
- rejected/blocked winners
- approved losers that matched known bad clusters
- missed trades with high posterior quality
- resize actions that improved or worsened utility
- inferior selection where a rejected alternative was materially better

Regret outputs should feed:
- `regret_case` learning rows
- policy adjustment evidence
- operator diagnostics
