# PRD: Adaptive Edge Estimation — v1.93

## Problem

The autonomous scan consistently produces `eligible: 0` because the edge calculation is
synthetic and disconnected from actual trading outcomes.

Current formula in `src/discovery/expressions.ts`:
```ts
expectedEdge = cluster.directionalBias === 'neutral' ? 0 : confidence * 0.1
```

Two compounding failures:

1. **Edge is not grounded in history.** `confidence * 0.1` is an arbitrary scaling of
   signal strength. Confidence ≠ edge. A signal can be strong and consistently wrong.
   The system has no way to learn this.

2. **Neutral bias → edge = 0.** Markets in a choppy or directionless state produce
   zero edge for all non-reflex hypotheses, regardless of volatility, crowding, or
   other signal inputs. This silently discards most market conditions.

Compound effect: the system generates no paper trades → no journal entries → no
empirical data → calibration gates remain idle → the system cannot self-improve.

There is also a classification bug: `_reflex` hypotheses are mapped to `'news_event'`
in `expressions.ts` (line 70) but should be `'liquidation_cascade'`. This routes
reflexivity trades through the news entry gate, which they fail due to absent news
trigger scores.

---

## Goal

Replace the synthetic edge formula with a **history-grounded, signal-modulated edge
estimate** that learns from the agent's trade journal over time.

The agent's confidence in a given setup should grow from its own experience of that
setup type (signal class × market regime × volatility × liquidity), not from an
arbitrary formula. Signal strength from discovery modulates the base estimate but
cannot manufacture edge the history does not support.

---

## How it works

### Phase 1 — Bootstrap (0 to `minSamples` trades in a segment)

Use a configurable `priorEdge` (default: 0.015). This is deliberately small — enough
to generate paper trades for data collection, not enough to size aggressively.

Signal strength from discovery acts as a multiplier (range 0.5–1.5×) around the
prior. A max-confidence signal gets the full prior; a zero-confidence signal gets half.

### Phase 2 — Empirical (≥ `minSamples` trades in a segment)

Compute `empiricalExpectancy = mean(capturedR)` over closed trades in the
`(signalClass, marketRegime, volatilityBucket, liquidityBucket)` segment.

Blend prior and empirical using sample-count weighting. As history grows, the
empirical estimate dominates.

Apply signal strength as a multiplier on the blended result.

### Phase 3 — Self-correcting

Signal classes with negative empirical expectancy naturally produce negative or
near-zero edge → fail `minEdge` → no trades → no further losses. The system
downtrades broken signals automatically without explicit blockers.

Signal classes with strong empirical expectancy get higher edge → more eligible
expressions → more trades → more data.

---

## Neutral bias fix

The `directionalBias === 'neutral' → edge = 0` rule is too broad. It makes sense for
`momentum_breakout` (which requires directional conviction) but not for:

- `mean_reversion`: profits from reversion regardless of direction; edge scales with
  vol regime and crowding, not directional bias
- `liquidation_cascade`: triggered by fragility conditions (funding, OI), not direction

The fix: apply the neutral-bias zero only to `momentum_breakout`. Other signal classes
fall through to the adaptive edge resolver.

---

## Bug fix included in this sprint

**`_reflex` → `'liquidation_cascade'`** (not `'news_event'`) in `expressions.ts`.
This is a prerequisite for the reflexivity signal to function correctly. It must ship
in the same sprint since it affects the edge and gate routing for the primary
crowding/fragility signal type.

## Success criteria

| Metric | Target |
|---|---|
| Paper trades generated per week (normal conditions) | ≥ 5 |
| Weeks to reach `minSamples` per major signal class | ≤ 8 |
| Edge estimate correlation with realised `capturedR` at 30+ samples | Pearson r ≥ 0.3 |
| System continues generating 0 trades for demonstrably negative segments | Required |
| `signalPerformance.minSharpe` gate becomes meaningful | At 20+ samples |

---

## Non-goals

- Backtesting on historical data (future sprint)
- Cross-asset correlation in edge estimation
- Real-money deployment (paper mode only for v1.93)
- Replacing `summarizeSignalPerformance` / `sharpeLike` gating (these remain as
  secondary guards; adaptive edge is the primary estimation path)

---

## Open questions

1. Should segment lookup fall back to coarser matches (e.g. ignore liquidityBucket) if
   the exact segment has insufficient samples? Or stay strict?
2. Should `capturedR` from paper trades carry the same weight as live trades? For now:
   yes, since we have no live trades.
3. Decay older trades? A `decayHalfLifeDays` config option is included but off by
   default until we have enough data to assess whether recency matters.
