# PRD: Adaptive Decision Learning — v2.2

## Problem

Thufir can now persist dossier-backed learning artifacts, but the current loop is still too weak to make him materially better at trading on its own.

What is proven today:
- a completed trade lifecycle can write a canonical dossier,
- `execution_quality` and `thesis_quality` can be stored separately,
- structured review and bounded counterfactual payloads can be persisted,
- similarity context can be injected into reasoning.

What is still missing:
- enough live dossier volume to create a useful corpus,
- review intelligence that goes beyond shallow heuristic labeling,
- counterfactuals strong enough to teach sizing and intervention quality,
- retrieval that materially changes entry and sizing decisions from real historical coverage,
- policy adaptation that acts on lessons instead of merely storing them.

The current loop is therefore mechanically alive but strategically underpowered.

## Goal

Ship the first genuinely adaptive decision-learning system for Thufir:

`decision -> dossier -> path -> review -> counterfactuals -> retrieval -> policy adaptation -> improved future decisions`

The architecture must be general, while the initial implementation remains specific to Thufir's trading context.

General architecture:
- canonical decision case
- structured context snapshot
- action and intervention records
- trajectory/path capture
- outcome and review records
- counterfactual set
- similarity retrieval
- policy adaptation

Thufir-specific payloads for v2.2:
- trading regime
- volatility/liquidity state
- entry stretch and timing quality
- catalyst freshness
- invalidation and time-stop quality
- gate intervention behavior
- portfolio crowding / overlap
- realized path quality
- proxy-vs-direct expression quality

## Product Thesis

Thufir becomes better not by storing more trades, but by learning better decision lessons.

The v2.2 system should improve:
- selectivity,
- sizing,
- leverage discipline,
- intervention quality,
- regime adaptation,
- transfer of prior mistakes into live choices.

It should not rely on raw PnL alone to determine whether a decision family is good.

## Current Repo Findings

The current repository already contains the critical pre-execution control seam, but it does not yet give learned evidence first-class authority there.

Observed current state:
- `src/core/autonomous.ts` already has the live decision chain:
  - proposal/discovery candidate
  - deterministic risk and wallet checks
  - `LlmEntryGate.evaluate(...)`
  - executor call
- `src/core/llm_entry_gate.ts` can currently:
  - approve
  - reject
  - resize
  - suggest leverage
- `src/core/autonomy_policy.ts` already provides deterministic policy gating and downweighting for some trade classes.
- `src/memory/learning_cases.ts` and `src/memory/learning_schema.ts` still only model a narrow case set centered on:
  - `comparable_forecast`
  - `execution_quality`
- retrieval, review, dossier linkage, intervention quality, regret learning, and adaptive policy evidence are not yet formal first-class runtime inputs in the current tree.

This means the core challenge for v2.2 is not inventing a place to wire learning back in. The place already exists. The challenge is giving learned evidence explicit, bounded authority at that decision seam.

## Scope

### 1. Canonical Decision Dossier Expansion

Upgrade `trade_dossiers` from a persistence shell into the canonical source of truth for the full decision lifecycle.

Every dossier must contain:
- entry context
- thesis and expected edge
- action request
- gate intervention request vs approval
- execution details
- path metrics
- close outcome
- structured review
- counterfactual set
- retrieval context used at decision time
- policy traces active at decision time

### 2. Live Dossier Volume and Coverage

Increase the rate and completeness of dossier creation for real production lifecycles.

This includes:
- every executed open must create or attach to a dossier,
- every reduce-only close must resolve the same dossier,
- every gate-approved and gate-resized trade must preserve intervention metadata,
- every full close must write review and learning records,
- skipped or blocked decisions must optionally create lightweight "non-executed decision cases" when useful for regret analysis.

### 3. Structured Review Intelligence

Introduce a stronger reviewer layer that evaluates a closed trade across multiple decision dimensions rather than a single win/loss label.

Required review axes:
- thesis quality
- context fit
- entry quality
- sizing quality
- leverage quality
- invalidation quality
- exit quality
- gate intervention quality
- portfolio interaction quality
- main success driver
- main failure mode
- repeat tags
- avoid tags
- review confidence

### 4. Strong Counterfactual Learning

Promote counterfactuals to a first-class learning object.

For each eligible closed trade, the system must evaluate bounded alternatives:
- no-trade baseline
- full requested size
- approved size
- alternate leverage cap
- delayed entry baseline
- early exit baseline
- invalidation exit baseline
- TTL exit baseline
- alternate expression baseline when available

The purpose is not perfect market replay. The purpose is to learn:
- whether the gate helped,
- whether the sizing was too large or too small,
- whether the expression choice was inferior,
- whether timing degraded an otherwise valid thesis.

### 5. Similarity Retrieval that Influences Decisions

Historical similarity must become a real input into live entry and sizing decisions.

Retrieval must operate on structured features, not only free text.

It must support:
- nearest wins
- nearest losses
- nearest cases where thesis was right but execution was bad
- nearest cases where thesis was wrong but execution was good
- nearest cases where gate intervention added value
- nearest cases with the same failure mode

Live decision paths must consume retrieval output explicitly, not only as incidental text decoration.

### 6. Policy Adaptation Engine

Add a policy layer that updates future trade constraints from accumulated dossier evidence.

Adaptation targets:
- setup-family max size
- regime-specific leverage caps
- confirmation requirements by catalyst class
- penalties for stretched entries
- penalties for weak analog support
- confidence haircuts under sparse precedent
- cooldowns after repeated failure modes
- intervention priors for resize / reject / cap decisions

Adaptation must be bounded, explainable, and reversible.

### 7. Path-Aware Learning

Store post-entry trajectory information so Thufir can learn from how trades evolved, not just how they ended.

Required path features:
- max favorable excursion
- max adverse excursion
- time to first confirmation
- time spent underwater
- time to invalidation
- whether thesis worked later after stopout
- whether a better timed entry was available shortly after
- whether path shape matched historical analogs

### 8. Regret and Opportunity Tracking

Teach Thufir not only from executed trades, but also from meaningful misses and interventions.

v2.2 must support:
- blocked trade that would have worked
- approved trade that should have been rejected
- resized trade where resize added value
- resized trade where resize cut too much
- no-trade where a high-quality opportunity was missed

This is essential for avoiding a timid or overfit policy.

### 9. Decision-Time Enforcement Contract

v2.2 must answer these three product questions explicitly:

1. Can retrieval really alter live decisions?
2. Can policy really resize or reject based on learned evidence?
3. Can weak evidence be prevented from poisoning the loop?

The answer must be "yes" in all three cases, but only through bounded, inspectable authority.

#### Retrieval authority

Retrieval is not allowed to remain prompt garnish.

Retrieval may:
- reduce confidence
- add explicit risk flags
- recommend size haircuts
- recommend leverage caps
- require stronger confirmation
- escalate a candidate into the reject band when historical support is materially weak and risk flags are strong

Retrieval may not:
- directly force execution
- override hard deterministic risk limits
- create permanent policy changes by itself
- bypass missing-data or weak-evidence controls

#### Policy authority

Policy adaptation is not advisory text. It is an enforcement layer.

Policy may:
- resize
- cap leverage
- impose confirmation requirements
- impose cooldowns
- reject a candidate when evidence-backed thresholds are crossed

Policy may not:
- exceed bounded deltas
- act on insufficient evidence
- persist forever without expiry and revalidation
- mutate silently without a decision trace

#### Evidence hygiene

Every learned signal must carry explicit evidence quality:
- evidence count
- similarity strength
- review confidence
- source quality
- freshness
- contradiction/disagreement score
- missing-data flags

Low-quality evidence must automatically downweight itself and must not create hard adaptation.

## Non-goals

1. Fine-tuning the base model.
2. Fully automated reinforcement learning from raw PnL.
3. Perfect historical simulation of every possible trade alternative.
4. Cross-domain implementation beyond trading in this release.
5. Removing existing human-readable rationale from the system.

## Design Principles

1. General architecture, specific payloads.
The learning loop must be reusable beyond trading, but v2.2 data fields should be optimized for Thufir's trading context.

2. Decision quality over outcome purity.
PnL matters, but it must not be the only truth signal.

3. Structured retrieval over prompt bloat.
The system should retrieve typed prior cases and summarized lessons, not dump raw history into prompts.

4. Bounded adaptation.
Policy changes must be gradual, attributable, and easy to inspect or roll back.

5. Evidence-linked learning.
Every learned lesson must link back to the dossier and the underlying trade path.

## User Stories

1. As an operator, I want every trade to produce a durable decision record so I can inspect why the system acted and what it later learned.
2. As an operator, I want the system to distinguish a good thesis with bad execution from a bad thesis with good execution.
3. As an operator, I want resize and reject decisions to be learnable so the gate becomes a better risk manager over time.
4. As an operator, I want similar historical dossiers to influence live entry decisions when relevant precedent exists.
5. As an operator, I want policy adaptation to become visible in future sizing and gating decisions.
6. As an operator, I want the architecture to stay general enough that the same learning pattern can later support cybersecurity or other domains.

## Success Criteria

| Metric | Target |
|---|---|
| Every executed open creates or attaches to a dossier | Required |
| Every full close on a dossier-backed trade writes review + thesis + execution learning | Required |
| Live retrieval uses structured similarity features and appears in decision traces | Required |
| Policy adaptation produces visible bounded deltas in future decision traces | Required |
| Counterfactual payload includes at least no-trade/full-size/approved-size/invalidation baselines | Required |
| Dossier corpus can segment outcomes by signal class + regime + failure mode | Required |
| Entry-time retrieval can distinguish similar wins vs similar losses | Required |
| The system can report resize value-add over a cohort | Required |

## Observability Requirements

The operator must be able to answer:
- How many dossier-backed trades were created this week?
- Which setup families are improving or degrading?
- Which gate interventions are helping?
- Which failure modes are repeating?
- Which policy adjustments are active right now?
- Which retrieved cases influenced a given decision?
- Whether the current policy is learning too quickly, too slowly, or from weak evidence.

## Risks

1. Sparse live data may cause premature overfitting.
2. Review intelligence may still be noisy if heuristics dominate.
3. Counterfactuals can become false precision if not bounded and labeled.
4. Retrieval can mislead if similarity features are weak or stale.
5. Policy adaptation can make the system timid if regret and missed-opportunity learning are underweighted.

## Release Decision

v2.2 is complete only when all five strategic gaps are closed in architecture and code:

1. more live dossier volume
2. better review intelligence
3. stronger counterfactuals
4. retrieval that influences entry decisions with meaningful historical coverage
5. policy adaptation that acts on stored lessons
