# v2.1 Learning Loop TDD

## Test Strategy

The v2.1 tests must prove that the learning loop now preserves and uses decision-level evidence across entry, gate, execution, and close.

The target is not subjective alpha evaluation. The target is deterministic proof that the system now stores, separates, retrieves, and scores the evidence required for later improvement.

## Capability Test Matrix

### 1. Canonical dossier persistence

Test:

- create open dossier with thesis, context, gate, execution, and exit-plan sections
- update same dossier through close with path, close, counterfactuals, and review
- verify merge behavior preserves prior sections

Expected proof:

- a single dossier row survives the full lifecycle
- linkage fields remain stable

### 2. Originator executed open emits full learning evidence

In `autonomous-wiring`:

- executed originator proposal creates canonical dossier
- executed originator proposal records executed non-reduce-only journal row
- comparable forecast learning case carries `source_dossier_id`
- thesis-quality learning case is written
- similarity retrieval summary is attached to dossier or learning context

### 3. Quant executed open emits the same learning contract

In autonomous quant path:

- executed quant trade creates canonical dossier
- comparable forecast and thesis-quality learning cases link to dossier
- journal evidence supports later close-time review

### 4. Similarity retrieval affects entry context

Unit/integration tests over retrieval layer:

- similar dossiers can be found by signal/regime/stretch features
- retrieval summary returns aggregate statistics and lesson tags
- entry-time caller receives retrieval output and persists it into dossier/gate context

### 5. Counterfactual scoring on close

In tool executor or lifecycle close path:

- a trade with requested size larger than approved size computes intervention counterfactuals
- dossier stores requested vs approved vs counterfactual outputs
- structured review marks whether the gate helped

Expected proof:

- resize/reject outcomes become first-class persisted evidence

### 6. Separate thesis and execution learning

Close path tests must verify:

- thesis-quality learning case exists
- execution-quality learning case exists
- the two can diverge in verdict/score
- positive PnL does not force thesis-quality and execution-quality to be identical

### 7. Structured reviewer output

Given a fully closed position:

- review persists `thesisVerdict`
- review persists `entryQuality`, `sizingQuality`, `leverageQuality`, `exitQuality`, `gateInterventionQuality`, `contextFit`
- review persists `mainSuccessDriver`, `mainFailureMode`, `lessons`, `repeatTags`, `avoidTags`

### 8. Backwards compatibility and failure tolerance

Regression tests must prove:

- legacy execution learning still persists
- missing optional retrieval data does not break entry execution
- close-time review persistence failures do not break trade close bookkeeping

## Acceptance Criteria

### A. Dossier lifecycle

Given an executed trade that later closes,
when the lifecycle completes,
then one dossier row contains canonical entry, close, counterfactual, and review data.

### B. Decision-time retrieval

Given a new trade candidate,
when the system evaluates entry,
then similar historical cases are retrieved and included in decision context.

### C. Intervention learning

Given a resized or rejected trade,
when enough close evidence exists,
then the dossier and review contain intervention scoring.

### D. Separate learning streams

Given a closed trade,
when learning cases are written,
then thesis-quality and execution-quality rows both exist and can differ.
