# v2.1 Learning Loop PRD

## Objective

Upgrade Thufir from a fragmented, outcome-led learner into a decision-led trading system that can improve entry selection, sizing, filtering, and trade management from its own history.

The system must be able to answer, for any materially executed trade:

- Was the thesis correct?
- Was the entry good or stretched?
- Was the requested size appropriate?
- Did the gate intervention help?
- Was the exit discipline good?
- What similar cases should affect the next decision?

## Core Problem

The current learning loop is too coarse and too late:

1. PnL is frequently treated as thesis truth.
2. Proposal, gate, execution, and close evidence are fragmented across unrelated tables.
3. Gate interventions are not scored against counterfactual outcomes.
4. Prior similar cases are not retrieved at entry time.
5. Post-trade reviews are not first-class structured lessons.

This makes the system vulnerable to learning the wrong lesson:

- profitable bad chases look correct
- disciplined small probes get confused with conviction trades
- resize/reject interventions get no credit
- past mistakes do not constrain current behavior

## Product Requirements

### 1. Canonical Trade Dossier

Every materially executed perp trade must have a canonical dossier that acts as the source of truth for learning.

Required coverage:

- identity: `id`, `symbol`, `status`, `direction`
- linkage: `source_trade_id`, `source_prediction_id`, `proposal_record_id`
- strategy metadata: `strategy_source`, `trigger_reason`, `execution_mode`
- timeline: `opened_at`, `closed_at`
- dossier payload sections:
  - `thesis`
  - `context`
  - `gate`
  - `execution`
  - `path`
  - `exit_plan`
  - `close`
  - `counterfactuals`
- structured review payload

### 2. Counterfactual Gate Scoring

The system must learn whether the gate helped.

For closed trades with sufficient evidence, persist counterfactual comparisons for:

- requested size vs approved size
- approved trade vs rejected trade baseline
- approved leverage vs capped leverage

Required outputs:

- intervention type
- realized outcome
- counterfactual outcome estimate
- intervention score
- whether the gate added or destroyed value

Counterfactuals may use deterministic heuristics in v2.1; they do not need full market replay simulation.

### 3. Separate Thesis Learning from Execution Learning

The learning loop must preserve separate judgments for:

- thesis quality
- entry quality
- sizing quality
- leverage quality
- exit quality
- gate intervention quality

The system must stop treating close PnL alone as the truth value of the thesis.

Required behavior:

- execution-quality learning cases must link to dossiers
- thesis-quality learning cases must persist independently from execution-quality cases
- close-time review must assign distinct verdicts even when realized PnL is positive

### 4. Similarity Retrieval at Entry Time

Before approving a trade, the system must retrieve similar historical dossiers/cases and inject the result into decision context.

Required matching dimensions:

- symbol class
- signal class
- trigger reason
- regime
- gate verdict history
- entry stretch metrics

Required outputs:

- top similar cases
- aggregate stats for similar cases
- prior lesson tags
- whether similar cases imply caution, approval, or size reduction

This retrieval must be available to both originator and gate-time decision logic.

### 5. Structured Post-Trade Reviewer

When a trade closes completely, the system must create a structured review that converts raw evidence into reusable lessons.

Required fields:

- `thesisVerdict`
- `entryQuality`
- `sizingQuality`
- `leverageQuality`
- `exitQuality`
- `gateInterventionQuality`
- `contextFit`
- `counterfactualNeeded`
- `mainSuccessDriver`
- `mainFailureMode`
- `lessons[]`
- `repeatTags[]`
- `avoidTags[]`

The reviewer may be heuristic in v2.1, but must emit stable structured outputs suitable for future policy adaptation.

## Non-Goals

v2.1 does not require:

- a fully autonomous self-tuning policy engine
- full market replay simulation for counterfactuals
- LLM-generated essay reviews

v2.1 must, however, create the data and runtime surfaces that make those upgrades straightforward.

## Success Criteria

For a resized momentum probe such as `XYZ:COIN`, the system must be able to recover:

- the original thesis
- requested size vs approved size
- entry stretch context
- retrieved similar-case summary used at decision time
- separate thesis and execution verdicts
- whether resize improved the outcome
- post-trade lessons that do not collapse to win/loss

## Acceptance Criteria

1. Every executed originator or quant open writes a canonical dossier.
2. Every full close updates that dossier with close summary, counterfactuals, and structured review.
3. Learning cases persist `source_dossier_id`.
4. Thesis-quality and execution-quality learning cases are both written and separately queryable.
5. Entry-time decision paths can retrieve similar historical cases and surface them in the decision context.
6. Gate interventions can be scored after close using persisted requested vs approved action data.

## Release Task Map

### Track A: Foundation and dossier model

- canonical dossier schema
- learning case linkage
- originator/quant open linkage

### Track B: Counterfactual and scoring model

- intervention scoring model
- thesis-learning vs execution-learning separation
- close-time score persistence

### Track C: Retrieval and review

- similarity search over dossiers/learning cases
- entry-time retrieval injection
- structured post-trade reviewer and lesson tags

## Risks

- backwards-compatibility drift on older DBs
- dossier duplication across multiple execution paths
- overfitting heuristic counterfactuals
- retrieval quality being too noisy without proper feature normalization
- increased close-time complexity causing missing reviews if not fail-safe

## Release Gate

v2.1 is acceptable when:

- all five capability areas exist in runtime and persistence
- targeted tests cover each capability
- regression tests across originator, tool executor, and close lifecycle pass
- production can demonstrate dossier creation, similarity retrieval, and structured review on real runtime data
