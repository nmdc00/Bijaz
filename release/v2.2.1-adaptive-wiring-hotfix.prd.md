# PRD: Adaptive Wiring Hotfix — v2.2.1

## Date

2026-05-15

## Problem

`v2.2` shipped the schema, retrieval helpers, review helpers, and adaptive trace scaffolding, but the runtime wiring is incomplete.

What production proves today:
- blocked autonomous decisions write `v2.2` dossier JSON
- `adaptive_pre_execution_trace` artifacts are live
- retrieval and policy trace content exists inside `dossier_payload`

What production does not prove:
- dedicated `retrieval_payload` and `policy_trace_payload` columns are populated
- full-close `v2.2` lifecycles persist `v2.2` close/review/counterfactual outputs
- typed similarity feature rows are written
- adaptive policy adjustments are generated or consumed from persistent evidence

Observed runtime gaps:
- autonomous dossier writes embed retrieval and policy trace in JSON but do not pass top-level `retrieval` / `policyTrace` into `upsertTradeDossier()`
- full-close path in `tool-executor` still writes dossier `version: 'v2.1'`
- close path persists `execution_quality` and `thesis_quality`, but not explicit counterfactual rows or typed similarity feature rows
- adaptive policy adjustment helpers and enforcer exist, but are not runtime-wired

## Goal

Ship `v2.2.1` as a wiring-correctness hotfix that makes the deployed adaptive learning loop internally consistent and production-verifiable.

This release is not a new feature expansion. It is a runtime completion pass for `v2.2`.

## Non-Goals

- no redesign of the learning architecture
- no speculative new policy domains beyond the existing `v2.2` contract
- no dashboard redesign
- no broad refactor of unrelated trading logic
- no branch-history repair work beyond normal release flow

## User / Operator Outcome

After `v2.2.1`:
- blocked and executed autonomous `v2.2` dossiers populate dedicated retrieval and policy-trace columns
- executed `v2.2` trades remain `v2.2` through full close
- full-close runtime persists explicit counterfactual rows and typed similarity features
- adaptive-policy persistence and consumption are no longer dead scaffolding
- production DB inspection can confirm the adaptive loop through first-class tables, not only nested dossier JSON

## Scope

### 1. Autonomous dossier write correctness

For every `v2.2` autonomous dossier write on blocked and open paths:
- persist the canonical dossier JSON
- persist dedicated `retrieval_payload`
- persist dedicated `policy_trace_payload`

This applies to:
- originator blocked
- originator executed open
- quant blocked
- quant executed open

### 2. Close-path version and section correctness

For every fully closed autonomous/perp lifecycle:
- keep dossier `version` at `v2.2`
- preserve earlier `v2.2` sections rather than downgrading or erasing them
- append/update close-path sections required for `v2.2`

### 3. Explicit counterfactual persistence

On full close:
- build bounded counterfactual records from existing close evidence
- persist rows into `trade_counterfactuals`
- keep a compact summary in the dossier JSON for quick inspection

### 4. Typed similarity feature persistence

On executed open and refresh on full close:
- derive typed similarity features from dossier/review state
- persist into `trade_similarity_features`

This is required so `v2.2` retrieval has a real feature corpus instead of relying only on dossier/review JSON scans.

### 5. Adaptive policy persistence and consumption

`v2.2.1` must wire the existing adaptive policy machinery into runtime in a bounded way.

Minimum requirement:
- derive policy adjustment proposals from eligible closed-trade evidence
- persist those proposals to `trade_policy_adjustments`
- consume active persisted adjustments at decision time
- include active adjustment ids in policy traces

Boundaries:
- only use the existing bounded proposal/enforcement model
- do not allow weak evidence to create hard policy rejects

### 6. Production-verifiable evidence

After release, operators must be able to confirm with direct DB checks:
- non-null `retrieval_payload`
- non-null `policy_trace_payload`
- at least one `trade_counterfactuals` row tied to a closed `v2.2` dossier
- at least one `trade_similarity_features` row tied to a `v2.2` dossier
- active or historical `trade_policy_adjustments` rows once evidence exists

## Acceptance Criteria

### Runtime correctness

- blocked autonomous `v2.2` dossiers populate dedicated retrieval and policy-trace columns
- executed autonomous opens populate dedicated retrieval and policy-trace columns
- full close does not overwrite `v2.2` dossier version with `v2.1`
- full close persists explicit counterfactual rows
- full close persists typed similarity feature rows
- active policy adjustments are read and surfaced in decision traces

### Testing

- targeted unit coverage exists for new wiring helpers and persistence bridges
- targeted integration coverage proves:
  - blocked path persistence
  - executed open persistence
  - full-close `v2.2` persistence
  - policy adjustment generation and consumption

### Production validation

- service healthy after deploy
- production DB shows live `v2.2.1` rows in the dedicated adaptive tables/columns

## Risks

- close-path merge behavior may unintentionally erase existing dossier sections if not patched carefully
- policy persistence can overfit if fed weak evidence or if activation/expiry rules are too loose
- similarity feature derivation can drift from dossier truth if derived from partially-populated sections

## Release Principle

`v2.2.1` should favor correctness and observability over ambition.

If a behavior cannot be proven through runtime tests and direct DB inspection, it should not be claimed complete.
